## 概念理解题

### Q1. Transformer 为什么取代了 LSTM？

**两个层面：**

1. **工程层面**：Transformer 可以并行计算，LSTM 必须一步一步顺序处理。并行意味着训练更快、能做更大的模型
2. **架构层面**：Transformer 的 self-attention 让任意两个词直接建立关联，不受距离限制。LSTM 的信息必须逐步传递，距离越远越容易丢失

只答"更快"只说了工程层面，漏了架构层面的优势。

---

### Q2. "Attention Is All You Need" 标题的含义

作者在说：我们不需要 LSTM 那些复杂的门控递推机制，只用一个更简单的 **self-attention** 机制就够了。

Self-attention 是什么：对序列中的每个 token，直接计算它和所有其他 token 的关联程度（权重），然后用这些权重来理解上下文。

标题的潜台词是"简化反而赢了"——因为简化换来了并行能力和规模优势。

---

### Q3. 什么是 emergent intelligence？

**Emergent intelligence（涌现智能）** = 当模型规模（参数量+训练数据）大到一定程度后，自动出现了训练时没有显式教过的能力。

为什么困惑：LLM 的训练目标只是"预测下一个最可能的 token"，这是一个纯统计任务。你预期它输出"看起来合理的文字"。但实际上它给出的经常是**正确的答案**——比如数学题的正确解、代码的正确实现。没有人训练它"做数学"，但足够大的规模让这个能力自己冒出来了。

我们理解 how（统计原理清楚），不完全理解 why（为什么统计预测做到极致就等于智能）。

---

### Q4. Context engineering vs prompt engineering

**Prompt engineering**：关注怎么写好一条指令（"请一步步思考""用专业口吻回答"）。

**Context engineering**：关注 LLM 看到的**整个输入**怎么构建，prompt 只是其中一部分。还包括：对话历史、RAG 检索到的文档、工具调用返回的数据、system prompt 等。

**具体场景**：做一个机票查询机器人。

- Prompt engineering 思维："请准确回答机票价格问题"
- Context engineering 思维：用户问"北京到东京多少钱"时，先调航班 API 拿到实时价格 ¥2,500，把价格数据塞进输入，再让模型基于这个数据回答

核心原则：LLM 只能基于它看到的输入来预测输出。你想要正确答案，相关信息就必须在输入里。

---

### Q5. Training time scaling vs inference time scaling

**Training time scaling（训练时扩展）**：更大的模型 + 更多训练数据 + 更多算力 → 模型本身更强。例子：从 GPT-3（175B）堆到 GPT-4（1.76T）。

**Inference time scaling（推理时扩展）**：模型不变，在使用时用技巧提升表现。例子：让模型先"思考"再回答（reasoning/CoT），或者用 RAG 往输入里塞更多参考信息。

**为什么说"正交"**：这两条路径互不依赖，可以同时做。一个大模型（training time scaling）加上 reasoning（inference time scaling），效果比单独做任何一个都好。两个维度独立地提升性能。

## 代码与实操题

### Q6. 第二次调用能答对吗？

**不能。** 每次 API 调用都是完全无状态的（stateless），第二次调用不知道第一次发生了什么。

修改方法——把对话历史手动带上：

```Python
messages = [
    {"role": "user", "content": "我叫小明"},
    {"role": "assistant", "content": "你好小明！"},  # 模拟第一次的回复
    {"role": "user", "content": "我叫什么？"}
]
openai.chat.completions.create(model="gpt-4.1-mini", messages=messages)
```

这就是 ChatGPT 网页版背后做的事——每次发消息都把完整聊天记录打包发给 API。

---

### Q7. 三种 role 分别是什么？

- **system**：系统指令，告诉模型它是谁、怎么回答（如"你是一个客服助手"）
- **user**：用户说的话
- **assistant**：模型之前回复过的话

写 `{"role": "assistant", "content": "你好！"}` **不是在让模型说话**，而是在告诉模型："上一轮你回复过这句话"。你在手动构造对话历史。模型不知道这是真的还是你编的——你甚至可以伪造对话历史，它照样当真。

## Token 与计费题

### Q8. 为什么用 token 而不是"一个单词一个输入"？

单词方案的问题：英语有几十万个词形（play, playing, played, player, plays, playful, replay...），每个都要占词表一个位置，词表太大。而且遇到没见过的词（如人名、新词）就完全不认识。

Token 方案的核心优势：**词根和词缀分开存，排列组合**。"play" + "ing/ed/er/s/ful" 等几个片段就覆盖了大量词形。像乐高积木，少量基础块拼出无数造型。GPT 词表大约 10 万个 token，就能表示任意文本。

---

### Q9. "exquisitely" vs "the" 的 token 数

- **"exquisitely"** ≈ 4 个 token（"ex" + "quis" + "ite" + "ly"），因为是罕见词，在训练数据中出现频率低
- **"the"** = 1 个 token，因为是英语中最常见的词，直接占一个词表位置

规律：**词越常见，token 数越少；词越罕见，拆得越碎。**

---

### Q10. 三位数加法准、四位数不行，和 token 的关系

GPT 词表里为每个三位数（000-999）都分配了一个 token，共 1000 个。

- **三位数加法**：两个操作数各是 1 个 token，结果也是 1 个 token。模型在训练数据中见过大量 "token A + token B = token C" 的模式，可以直接"记住"
- **四位数加法**：操作数跨了多个 token（如 1234 → "12" + "34"），模型不能简单记忆，需要真正理解进位逻辑，这对早期模型来说太难了

---

## 架构与成本题

### Q11. 从输入到收到回复的完整步骤

1. 你的代码把 messages 发送给 API 服务器
2. Tokenizer 把所有文字拆成 token ID 序列
3. Token ID 序列喂进 Transformer
4. Transformer 通过多层 self-attention 处理（浅层识别语法 → 中层理解语义 → 深层综合推理）
5. 输出概率分布，选出最可能的下一个 token
6. 把新 token 拼回序列末尾，再跑一遍 Transformer（循环）
7. 重复步骤 5-6，直到输出 EOS（结束标记）或达到 max_tokens 上限
8. 把所有生成的 token 解码回文字，返回给你的代码

关键理解：模型不是一次性生成整段回复，而是**一个 token 一个 token 往外蹦**。这就是为什么 ChatGPT 的回复是逐字出现的。

---

### Q12. 聊了 50 轮后第 51 轮的成本

**Input 包含什么**：前 50 轮的所有 user 消息 + 所有 assistant 回复 + system prompt + 你第 51 轮的新消息。全部打包发给 API。

**成本关系**：聊得越久，每次发送的 input token 越多，每次调用越贵。成本是累积增长的，不是固定的。

**省钱办法**：只发最近几轮对话，不带完整历史。**代价**：模型会"失忆"，不记得早期聊过的内容，回答质量可能下降。

---

### Q13. Context window 1M vs 200K 的实际意义

1M token ≈ 整本莎士比亚全集，200K ≈ 大约一本中等长度的书。

**差距最重要的场景**：

- 需要一次性处理大量文档的任务（如"帮我分析这整本书""总结这 100 页的报告"）
- RAG 场景中需要往输入里塞大量检索结果
- 超长对话（几百轮之后 200K 可能不够用，1M 还有余量）

日常短对话中这个差距感知不大，都够用。

## 判断题

### Q14. "参数量越大的模型一定越强。"

**错。** Gemma 270M 比 GPT-2 1.5B 强，因为训练数据质量/数量、训练方法（RLHF、知识蒸馏）、架构微调都在进步。参数量是重要因素但不是唯一因素。同时代的模型之间，更多参数通常更强；但跨时代比较，小模型完全可以超越旧的大模型。

---

### Q15. "ChatGPT 能记住你之前说过的话，因为它有内置的记忆功能。"

**错。** 每次 API 调用都是无状态的。ChatGPT 的"记忆"是前端把完整对话历史打包重新发送给模型实现的，不是模型本身有记忆。模型只是在读到历史对话后，基于这些输入预测输出。

---

### Q16. "Reasoning 模型的思考过程你看不到，但你不需要为这些 token 付费。"

**错。** 思考过程的 token 你看不到，但**要付费**。这些是 output token，计算确实发生了，你需要为这些计算买单。这也是 reasoning 模型成本不太可预测的原因之一。
